## 🟡 Important

### Q1 🟡 `config_path()` silently swallows `current_dir()` errors

**Where**
`packages/agent-adapter/src/cli.rs:49` and `packages/agent-adapter/src/cli.rs:55`

**Code**
```rust
fn config_path() -> std::path::PathBuf {
    std::env::current_dir()
        .unwrap_or_default()
        .join(AGENT_DIR)
        .join(CONFIG_FILE)
}
```

**Why**
`std::env::current_dir()` returns `Err` when the current working directory no longer exists (e.g., deleted after process start). `unwrap_or_default()` silently produces `PathBuf("")`, causing files to be written relative to the process start directory with no error indication. The same pattern repeats in `artifacts_path()`.

**Fix**
Propagate the error via `?` (changing return type to `Result<PathBuf, AdapterError>`) or replace `unwrap_or_default()` with an explicit error that mentions the I/O failure mode.

### Q2 🟡 `init_profile()` leaves partial state when config write fails

**Where**
`packages/agent-adapter/src/cli.rs:72` and `packages/agent-adapter/src/cli.rs:80`

**Code**
```rust
std::fs::create_dir_all(&agent_dir)?;
std::fs::create_dir_all(artifacts_path())?;
let yaml = serde_yaml::to_string(&config)?;
std::fs::write(&config_path, yaml)?;
```

**Why**
Directories are created before the config file is written. If the `fs::write` call fails (disk full, permission denied), empty `.agent/` and `.agent/adapter_artifacts/` directories remain. While `create_dir_all` is idempotent for retries, the partial state is user-visible and unexpected. More importantly, on retry from a different CWD, the same problem as Q1 arises.

**Fix**
Write the YAML config to a temporary file within `.agent/`, then atomic-rename it to the final path. On error, remove the temp file. This keeps the operation self-contained.

## 💭 Discussion

### Q6 💭 Input validation for config fields is deferred

**Where**
`packages/agent-adapter/src/config.rs:68–177`

**Why**
Config fields accept any string/integer value during deserialization. Specifically:
- `graph_engine_binary_path` defaults to `"./target/release/stage"` which is only correct for release builds; debug builds produce `./target/debug/stage`.
- `max_task_minutes: 10` has no upper bound
- `max_copied_artifact_bytes` and `max_total_copied_bytes` have no validation
- `allowed_policy` strings have no schema enforcement

These are reasonable to defer, but `graph_engine_binary_path` will break developer workflows. Consider resolving it at runtime (e.g., detect debug vs release via `cfg!(debug_assertions)`).

### Q7 💭 Profile name empty check only in `validate_profile()`, not at parse time

**Where**
`packages/agent-adapter/src/cli.rs:108–115`

**Why**
`validate_profile()` manually checks for empty profile names and empty runtimes, but these checks are not enforced during YAML deserialization. A directly-created config could bypass these validations.
